# エージェント・マルチエージェント・タスク管理・Worktree 関連機能 分析レポート

対象バージョン: 2.0.73 〜 2.1.76
抽出基準: エージェント、サブエージェント、マルチエージェント、タスク管理、worktree 隔離、バックグラウンドタスク、プランモード、リモートコントロール関連

---

## 採点基準

| 評価 | 意味 |
|------|------|
| ★★★★★ | ゲームチェンジャー。ワークフローを根本的に変える |
| ★★★★☆ | 非常に重要。多くのユーザーに大きな影響 |
| ★★★☆☆ | 重要。特定のユースケースで価値が高い |
| ★★☆☆☆ | 便利。あると嬉しい改善 |
| ★☆☆☆☆ | マイナー。特定の状況でのみ有用 |

## 抽出・採点結果

---

### 1. エージェントチーム機能（リサーチプレビュー）

**バージョン:** 2.1.32

**機能の概要:**
複数の Claude Code エージェントがチームとして協調動作するリサーチプレビュー機能。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 環境変数で有効化。

**採点:** ★★★★★

**評価の理由:**
マルチエージェント協調という概念そのものを Claude Code に持ち込む、アーキテクチャレベルの変革。単一エージェントの限界（コンテキスト長、並列処理不可）を突破し、大規模・複雑なタスクを分散処理できるようになる。リサーチプレビューの段階であっても、将来の正式リリースで業界標準になりうるポテンシャルがある。

**具体的なユースケース例:**
- 大規模リファクタリングをフロントエンド担当・バックエンド担当・テスト担当の3エージェントに分担
- 長大なドキュメント解析を複数エージェントで並列処理してレポートを統合
```
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 claude
```

### 2. `--worktree` フラグによる隔離 git worktree 起動

**バージョン:** 2.1.49

**機能の概要:**
`--worktree` (`-w`) フラグで Claude を独立した git worktree 内で起動し、メインブランチへの影響を隔離しながら作業可能。

**採点:** ★★★★★

**評価の理由:**
並列開発・実験的変更・マルチエージェント協調において「安全な隔離空間」を提供する根幹機能。複数のエージェントが同一リポジトリで衝突せずに同時作業できるようになり、CI/CD パイプラインとの統合も容易になる。エージェントチーム機能と組み合わせることで真価を発揮する。

**具体的なユースケース例:**
- フィーチャーブランチを汚染せずに AI による実験的なリファクタリングを試す
- 複数エージェントがそれぞれ独立した worktree で並列タスクを処理
```bash
claude --worktree -p "認証モジュールをリファクタリングして"
claude -w -p "テストスイートを追加して"
```

### 3. 新タスク管理システム（依存関係トラッキング）

**バージョン:** 2.1.16

**機能の概要:**
依存関係トラッキングを含む新しいタスク管理システムを導入。タスク間の依存関係を定義・追跡することで、複雑なワークフローの順序制御が可能。

**採点:** ★★★★★

**評価の理由:**
単純なタスクリストから「依存関係を持つワークフローグラフ」へ進化したことで、エージェントが複雑なプロジェクト計画を自律的に実行できるようになる。マルチエージェント環境において各サブエージェントのタスク間依存を管理する基盤となり、エージェントオーケストレーションの品質を根本的に引き上げる。

**具体的なユースケース例:**
- 「DB スキーマ作成 → API 実装 → テスト作成 → ドキュメント生成」という依存連鎖をエージェントが自律管理
- ビルド失敗時に依存タスクをスキップし、失敗ツリーを可視化

### 4. `TeammateIdle` / `TaskCompleted` フックイベント

**バージョン:** 2.1.33

**機能の概要:**
マルチエージェントワークフロー向けに `TeammateIdle`（チームメイトがアイドル状態）と `TaskCompleted`（タスク完了）のフックイベントを追加。

**採点:** ★★★★☆

**評価の理由:**
マルチエージェント間の状態同期・連携制御を外部スクリプトやシステムから行える重要な拡張点。エージェントAの完了をトリガーにエージェントBを起動するといった動的なオーケストレーションが実現できる。エージェントチーム機能を実用レベルで使うには不可欠なプリミティブ。

**具体的なユースケース例:**
- `TaskCompleted` フックで Slack 通知を送信
- `TeammateIdle` フックで次のサブタスクをキューから取り出して割り当て
```json
{
  "hooks": {
    "TaskCompleted": "python3 notify_team.py"
  }
}
```

### 5. エージェント定義での `isolation: worktree` サポート

**バージョン:** 2.1.50

**機能の概要:**
エージェント定義ファイルのフロントマターに `isolation: worktree` を指定することで、そのエージェントが自動的に隔離された worktree 内で動作するよう設定できる。

**採点:** ★★★★☆

**評価の理由:**
エージェントチームにおける「各エージェントの安全な作業空間の自動確保」を宣言的に設定できる。手動で worktree を管理する必要がなくなり、エージェント定義だけで隔離ポリシーを表現できる。エージェントチームのスケールアップに直接貢献する。

**具体的なユースケース例:**
- エージェント YAML に `isolation: worktree` を追記するだけで、メインブランチを守りながら並列エージェントを安全に運用
```yaml
---
name: feature-dev-agent
isolation: worktree
---
新機能を実装してください。
```

### 6. `WorktreeCreate` / `WorktreeRemove` フックイベント

**バージョン:** 2.1.50

**機能の概要:**
worktree の作成・削除タイミングでフックを実行可能。外部システムへの通知やクリーンアップ処理を自動化できる。

**採点:** ★★★☆☆

**評価の理由:**
worktree ライフサイクルを外部システムと統合するための重要な拡張点。CI パイプラインや監視システムと連携する際に有用だが、worktree を積極的に活用していないユーザーには関係が薄い。エージェントチーム基盤の整備として位置づけられる。

**具体的なユースケース例:**
- `WorktreeCreate` フックで GitHub PR ドラフトを自動作成
- `WorktreeRemove` フックで対応するリモートブランチをクリーンアップ
```json
{
  "hooks": {
    "WorktreeCreate": "gh pr create --draft --title 'WIP: Agent task'",
    "WorktreeRemove": "git push origin --delete $BRANCH_NAME"
  }
}
```

### 7. `worktree.sparsePaths` 設定（大規模モノレポ対応）

**バージョン:** 2.1.76

**機能の概要:**
`claude --worktree` 使用時に git sparse-checkout を活用し、大規模モノレポで必要なディレクトリのみをチェックアウトできる設定。

**採点:** ★★★☆☆

**評価の理由:**
モノレポ環境での worktree 活用における実用上のボトルネック（全ファイルチェックアウトによる遅延・容量問題）を解消する実用的な改善。ただし、大規模モノレポを扱うチームに限定される。そうした環境では worktree 機能の採用を大きく後押しする。

**具体的なユースケース例:**
- 数万ファイルのモノレポで `packages/auth` と `packages/api` のみをチェックアウトして AI タスクを実行
```json
{
  "worktree": {
    "sparsePaths": ["packages/auth", "packages/api", "shared/utils"]
  }
}
```

### 8. `claude agents` CLI コマンド

**バージョン:** 2.1.50

**機能の概要:**
設定済みエージェントを CLI から一覧表示するコマンド。

**採点:** ★★☆☆☆

**評価の理由:**
エージェント管理の利便性を高める補助的なコマンド。エージェント数が増えてくると重要度が上がるが、単体では大きなワークフロー変化をもたらさない。エージェントチーム機能の運用ツールとして位置づけられる。

**具体的なユースケース例:**
- チームで共有しているエージェント一覧を素早く確認
```bash
claude agents
# => frontend-agent, backend-agent, test-agent, docs-agent
```

### 9. `Task(agent_type)` 構文でスポーン可能なサブエージェントを制限

**バージョン:** 2.1.33

**機能の概要:**
`Task(agent_type)` 構文により、特定のエージェントタイプのみをサブエージェントとしてスポーン可能に制限できる。セキュリティ・スコープ管理に活用。

**採点:** ★★★☆☆

**評価の理由:**
マルチエージェント環境における「最小権限の原則」を実現する重要なセキュリティ機能。意図しないエージェントのスポーンを防ぎ、エージェント間の役割・責任を明確に区切ることができる。エージェントチームが本番運用に近づくほど重要性が増す。

**具体的なユースケース例:**
- テストエージェントがデプロイエージェントをスポーンできないよう制限
```yaml
allowed-tools:
  - Task(test-agent)
  - Task(lint-agent)
```

### 10. `Task(AgentName)` 構文で特定のエージェントを無効化

**バージョン:** 2.1.0

**機能の概要:**
`Task(AgentName)` 構文で特定の名前を持つエージェントを無効化（ブロック）できる。

**採点:** ★★☆☆☆

**評価の理由:**
特定エージェントの一時無効化という管理操作を宣言的に行える便利機能。エージェントチームの運用・デバッグ時に有用だが、日常的に使用する機能ではない。

**具体的なユースケース例:**
- 問題のあるエージェントをコード変更なしに一時無効化してデバッグ
```yaml
disabled-agents:
  - Task(experimental-agent)
```

### 11. エージェントフロントマターでのフックサポート

**バージョン:** 2.1.0

**機能の概要:**
エージェント定義のフロントマターで `PreToolUse`, `PostToolUse`, `Stop` フックを設定可能。エージェント固有のライフサイクル制御が実現。

**採点:** ★★★★☆

**評価の理由:**
エージェントごとに独立したフック設定を持てることで、各エージェントの振る舞いを細かく制御・カスタマイズできる。マルチエージェント環境では「エージェントAはツール使用前に承認必須」「エージェントBは停止時に結果をログに記録」といった個別ポリシーを宣言的に表現できる。

**具体的なユースケース例:**
- デプロイエージェントの `PreToolUse` フックで必ず人間の承認を要求
- レポートエージェントの `Stop` フックで成果物を自動的に S3 にアップロード
```yaml
---
name: deploy-agent
hooks:
  PreToolUse: "require-approval.sh"
  Stop: "upload-artifacts.sh"
---
```

### 12. `エージェントに memory フロントマターフィールドを追加`

**バージョン:** 2.1.33

**機能の概要:**
エージェント定義に `memory` フロントマターフィールドを追加。`user`, `project`, `local` の3スコープでメモリを管理。

**採点:** ★★★☆☆

**評価の理由:**
エージェントが実行をまたいで状態・知識を保持できることで、継続的なタスク遂行能力が大幅に向上する。マルチエージェント環境では「どのエージェントがどのスコープのメモリにアクセスできるか」を制御する基盤となる。

**具体的なユースケース例:**
- プロジェクトスコープのメモリで「このエージェントはコーディング規約を常に記憶」
```yaml
---
name: code-review-agent
memory: project
---
```

### 13. `フックイベントに agent_id・agent_type を追加`

**バージョン:** 2.1.69

**機能の概要:**
フックイベントの入力データに `agent_id`（サブエージェント用）と `agent_type`（サブエージェントおよび `--agent` 用）フィールドを追加。

**採点:** ★★★☆☆

**評価の理由:**
マルチエージェント環境でどのエージェントがイベントを発火したかをフック側で識別できるようになる、重要なトレーサビリティ改善。ログ・監査・デバッグのほか、エージェント別に異なるフック挙動を実装するために不可欠。

**具体的なユースケース例:**
- フックスクリプトで `agent_type` を判定し、デプロイエージェントの操作のみ Slack に通知
```python
import json, sys
event = json.load(sys.stdin)
if event.get("agent_type") == "deploy-agent":
    notify_slack(event)
```

### 15. `ExitWorktree` ツール

**バージョン:** 2.1.72

**機能の概要:**
`EnterWorktree` セッションから離脱するための `ExitWorktree` ツールを追加。worktree への入退室をプログラマチックに制御可能。

**採点:** ★★★☆☆

**評価の理由:**
`EnterWorktree` / `ExitWorktree` のペアにより worktree のスコープ管理が完結する。エージェントが自律的にworktree を切り替えながら複数の作業コンテキストを扱えるようになり、複雑なマルチブランチ作業フローを自動化できる。

**具体的なユースケース例:**
- エージェントが feature-A の worktree で作業 → `ExitWorktree` → feature-B の worktree へ遷移、という自律的なブランチ切り替えワークフロー

### 16. `/plan` コマンドショートカット（引数付き）

**バージョン:** 2.1.0 / 2.1.72

**機能の概要:**
`/plan` コマンドでプロンプトからプランモードに素早く入れるショートカット。2.1.72 では説明引数を直接指定可能（例: `/plan fix the auth bug`）。

**採点:** ★★★☆☆

**評価の理由:**
プランモードへの導線を大幅に短縮し、「まず計画してから実行」というベストプラクティスを日常的に使いやすくする。特に複雑なエージェントタスクの前段階として計画を立てるワークフローの定着に貢献。引数サポートにより一段とスムーズになった。

**具体的なユースケース例:**
```
/plan fix the auth bug
/plan migrate database schema to v2
```

### 17. `claude remote-control` サブコマンド

**バージョン:** 2.1.51

**機能の概要:**
外部ビルドや自動化ツールからローカルの Claude Code セッションをリモート制御できるサブコマンドを全ユーザーに提供。

**採点:** ★★★★☆

**評価の理由:**
CI/CD パイプライン・外部スクリプト・他ツールから Claude Code を操作できるようになり、Claude Code を「外部から呼び出せる AI エンジン」として位置づけられる。エージェントの自動化・オーケストレーション基盤として重要な拡張。特に headless 環境での活用可能性が高い。

**具体的なユースケース例:**
- GitHub Actions から `claude remote-control` でコードレビューを自動実行
- ビルドスクリプトからエラー発生時に Claude Code を呼び出して自動デバッグ
```bash
claude remote-control --name ci-session "テスト失敗を修正して"
```

### 19. Ctrl+B でバックグラウンド bash コマンドとエージェントを統一的にバックグラウンド化

**バージョン:** 2.1.0

**機能の概要:**
Ctrl+B のショートカットで bash コマンドとエージェントの両方を統一的にバックグラウンドへ移行できる。

**採点:** ★★★☆☆

**評価の理由:**

**具体的なユースケース例:**

### 22. セッション内クロンスケジューリングツールと `/loop` コマンド

**バージョン:** 2.1.71

**機能の概要:**
`/loop` コマンドで定期的にプロンプトやスラッシュコマンドを実行可能（例: `/loop 5m check the deploy`）。合わせてセッション内クロンスケジューリングツールも追加。

**採点:** ★★★★☆

**評価の理由:**
エージェントによる定期的な自律モニタリング・チェックを実現するゲームチェンジング的な機能。CI の代替・監視エージェント・定期レポート生成など、エージェントを「常駐型の自律プロセス」として運用できるようになる。バックグラウンドエージェントと組み合わせることで特に強力。

**具体的なユースケース例:**
- デプロイ後に5分ごとにヘルスチェックを監視
- 1時間ごとにコードベースの品質指標をレポート
```
/loop 5m check the deploy status and report any errors
/loop 1h generate a code quality summary
```

### 23. `CLAUDE_CODE_DISABLE_CRON` 環境変数

**バージョン:** 2.1.72

**機能の概要:**

## 全体的な考察

2.1.x 系のリリースは **マルチエージェント・worktree による並列分散作業** を中核テーマとして据えていることが明確に読み取れる。特に以下の3つの機能群が互いに補完し合う設計になっている。

1. **隔離基盤**: `--worktree`、`isolation: worktree`、`worktree.sparsePaths` が「安全な作業空間」を提供
2. **協調制御**: エージェントチーム、`TeammateIdle`/`TaskCompleted` フック、`agent_id` 識別が「複数エージェントの連携」を実現
3. **自律実行**: 新タスク管理（依存関係トラッキング）、`/loop`、`remote-control` が「人手を介さない継続実行」を可能にする

これら3つが揃ったことで、Claude Code は単なる「対話型 AI アシスタント」から **「自律的なソフトウェアエンジニアリングチームのオーケストレーター」** へと進化しつつある段階にあると言える。
