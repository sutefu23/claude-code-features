### 1. Bash パーミッションルールのワイルドカードサポート

**バージョン:** 2.1.0

**機能概要:**
`Bash(npm *)` や `Bash(git * main)` のようにワイルドカードパターン `*` を使って、特定のコマンドプレフィックスやパターンに一致するBashツール実行を一括承認できるようになった。

**評価:** ★★★★★

**理由:**
パーミッション管理の柔軟性が飛躍的に向上し、従来のコマンド単位の個別許可から、コマンドファミリー単位の包括的な許可へと移行できる。セキュリティを保ちながら、頻繁な承認ダイアログの手間を大幅に削減できる。CI/CD環境やチーム共有設定での運用効率が根本的に変わる。

**具体的なユースケース例:**
```json
// settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git * main)",
      "Bash(docker compose *)"
    ]
  }
}
```
- npm run build / npm run test / npm install など npm コマンド全般を一括承認
- `git push origin main` や `git merge main` など main ブランチ操作を一括承認

### 2. modelOverrides 設定

**バージョン:** 2.1.73

**機能概要:**
モデルピッカーのエントリを、Bedrock 推論プロファイル ARN 等のカスタムプロバイダーモデル ID にマッピングできる設定。企業独自のモデルエンドポイントやプロバイダーに対応可能。

**評価:** ★★★★★

**理由:**
AWS Bedrock や Azure OpenAI などのエンタープライズ向けクラウドプロバイダーを使っている組織が、Claude Code のUIから自社のカスタムエンドポイントをシームレスに利用できるようになる。ベンダーロックインを避けつつ、ガバナンス要件（データローカリティ、監査等）を満たしながら Claude Code を活用できる点が非常に重要。

**具体的なユースケース例:**
```json
// settings.json
{
  "modelOverrides": {
    "claude-opus-4-5": "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-opus-4-5-20251101-v1:0",
    "claude-sonnet-4-5": "arn:aws:bedrock:us-west-2::foundation-model/anthropic.claude-sonnet-4-5-20250929-v1:0"
  }
}
```
- AWS Bedrock の推論プロファイルを組織の標準モデルとして設定
- データが特定リージョンから出ないことを保証するエンタープライズ要件への対応

### 3. claude auth サブコマンド群

**バージョン:** 2.1.41

**機能概要:**
`claude auth login`、`claude auth status`、`claude auth logout` の3つのCLIサブコマンドを追加。認証状態の管理がコマンドラインから完結できるようになった。

**評価:** ★★★★☆

**理由:**
CI/CD パイプラインやスクリプト化された環境での認証自動化が可能になり、ヘッドレス環境での運用が大幅に改善される。認証状態の確認・切り替えをスクリプトに組み込める点は、複数アカウントを使い分けるチームや自動化を重視する組織に特に価値が高い。

**具体的なユースケース例:**
```bash
# CI環境での認証確認
claude auth status

# 本番環境と開発環境のアカウント切り替え
claude auth logout
claude auth login

# スクリプトでの認証チェック
if ! claude auth status > /dev/null 2>&1; then
  claude auth login
fi
```

### 4. Opus 4.6 1M コンテキストウィンドウのデフォルト有効化

**バージョン:** 2.1.75

**機能概要:**
Opus 4.6 モデルで 1M（100万トークン）コンテキストウィンドウが Max、Team、Enterprise プランのユーザーに対してデフォルトで有効化された。

**評価:** ★★★★★

**理由:**
大規模コードベースの解析や、長大なドキュメントの一括処理が根本的に変わる。従来は複数セッションに分割していた作業を単一セッションで完結できるようになり、コンテキスト喪失によるミスや繰り返し説明のコストが大幅に削減される。エンタープライズユーザーにとっては特にゲームチェンジャー。

**具体的なユースケース例:**
- 数十万行規模のモノレポを一括でコンテキストに入れてリファクタリング計画を立案
- 数百ページの仕様書と実装コードを同時に参照してギャップ分析
- `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` で無効化（コスト管理やレイテンシ最適化のため）

### 5. CLAUDE_CODE_DISABLE_1M_CONTEXT 環境変数

**バージョン:** 2.1.50

**機能概要:**
1M コンテキストウィンドウをオプトアウトするための環境変数。コスト管理やレイテンシ最適化のために、組織やユーザー単位で 1M コンテキストを無効化できる。

**評価:** ★★★☆☆

**理由:**
1M コンテキストが有効になった（2.1.75）ことと合わせて考えると、コスト統制が必要なエンタープライズ環境での必須オプションになる。特定のプロジェクトやユーザーに対してコンテキストサイズを制限することで、APIコストの予測可能性を確保できる。

**具体的なユースケース例:**
```bash
# コスト管理のため 1M コンテキストを無効化
export CLAUDE_CODE_DISABLE_1M_CONTEXT=1

# CI環境の .env ファイルに設定
CLAUDE_CODE_DISABLE_1M_CONTEXT=1
```

### 7. language 設定

**バージョン:** 2.1.0

**機能概要:**
`settings.json` に `language` 設定を追加することで、Claude の応答言語を指定できる。例: `language: "japanese"` で日本語応答に固定。

**評価:** ★★★★☆

**理由:**
非英語圏ユーザーにとって、毎回「日本語で答えてください」と指示する必要がなくなり、ワークフローが大幅に改善される。チームや組織単位でプロジェクト設定に言語を固定することで、応答の一貫性が保たれる。グローバルチームでの多言語運用や、ローカライズされた開発環境の構築が容易になる。

**具体的なユースケース例:**
```json
// ~/.claude/settings.json または .claude/settings.json
{
  "language": "japanese"
}
```
- 日本語開発チームのプロジェクトでデフォルト応答言語を日本語に設定
- 組織の settings.json に言語設定を含めて全メンバーに配布

### 8. sandbox.enableWeakerNetworkIsolation 設定

**バージョン:** 2.1.69

**機能概要:**
macOS 限定で、MITM プロキシを使用する環境で `gh`、`gcloud`、`terraform` 等のコマンドの TLS 証明書検証を許可する設定。企業のネットワーク監視環境での利用を可能にする。

**評価:** ★★★☆☆

**理由:**
企業のプロキシ環境（SSL インスペクションを行うファイアウォール等）では、これがないと多くの CLIツールが証明書エラーで動作しない。エンタープライズ環境への展開において重要なブロッカーを解消する機能だが、セキュリティリスクも伴うため慎重な判断が必要。

**具体的なユースケース例:**
```json
// .claude/settings.json
{
  "sandbox": {
    "enableWeakerNetworkIsolation": true
  }
}
```
- 企業のSSLインスペクションプロキシ環境で `gh` コマンドを正常動作させる
- `NODE_EXTRA_CA_CERTS` と組み合わせた企業CA証明書の信頼設定

### 9. includeGitInstructions 設定 / CLAUDE_CODE_DISABLE_GIT_INSTRUCTIONS 環境変数

**バージョン:** 2.1.69

**機能概要:**
ビルトインのコミット・PR ワークフロー指示をシステムプロンプトから除外できる設定と環境変数。独自の Git ワークフロー規約を持つ組織が、デフォルト指示の干渉を防ぎカスタム CLAUDE.md の指示を優先させられる。

**評価:** ★★★☆☆

**理由:**
独自のブランチ戦略やコミットメッセージ規約（Conventional Commits 等）を厳密に適用したいチームにとって、デフォルト指示の上書きは長年の課題だった。CLAUDE.md のカスタム指示とデフォルト指示が競合する問題を根本的に解決できる。

**具体的なユースケース例:**
```json
// .claude/settings.json
{
  "includeGitInstructions": false
}
```
```bash
# 環境変数での制御
export CLAUDE_CODE_DISABLE_GIT_INSTRUCTIONS=1
```
- 独自のコミットメッセージフォーマット（Jira チケット番号必須等）を CLAUDE.md で定義し、デフォルト指示の干渉を排除
- モノレポで複数プロジェクトごとに異なる Git ワークフローを適用

### 10. respectGitignore 設定

**バージョン:** 2.1.0

**機能概要:**
`settings.json` でプロジェクト単位に `@-mention` ファイルピッカーが `.gitignore` に従うかどうかを制御できる設定。

**評価:** ★★☆☆☆

**理由:**
デフォルト動作で `.gitignore` を尊重するかどうかの制御は、特定のプロジェクト構成（生成ファイルや除外設定が特殊な場合）で必要になる。多くのユーザーには影響が少ないが、特定のセキュリティ要件（機密ファイルを誤って参照しないようにする）や大規模プロジェクトでのノイズ低減に価値がある。

**具体的なユースケース例:**
```json
// .claude/settings.json
{
  "respectGitignore": false
}
```
- ビルド成果物を意図的に参照させたい場合に `false` に設定
- node_modules 内の特定ファイルを分析したい場合

### 11. autoMemoryDirectory 設定

**バージョン:** 2.1.74

**機能概要:**
Claude のオートメモリ（会話から自動生成されるメモリ）の保存ディレクトリをカスタマイズできる設定。デフォルト以外の場所にメモリファイルを配置できる。

**評価:** ★★☆☆☆

**理由:**
チームでメモリファイルを共有リポジトリで管理したい場合や、特定のディレクトリ構成を持つプロジェクトでデフォルトパスが不適切な場合に有用。ただし、影響を受けるユーザー層は限定的。

**具体的なユースケース例:**
```json
// .claude/settings.json
{
  "autoMemoryDirectory": "/path/to/shared/team-memory"
}
```
- チームの共有メモリをバージョン管理対象ディレクトリに配置
- プロジェクトごとに独立したメモリディレクトリを管理

### 12. plansDirectory 設定

**バージョン:** 2.1.9

**機能概要:**
プランファイルの保存先ディレクトリをカスタマイズできる設定。デフォルト以外の場所にプランファイルを保存可能。

**評価:** ★★☆☆☆

**理由:**
プランファイルをプロジェクトの特定ディレクトリ（例: `docs/plans/`）に整理したいチームや、バージョン管理に含めたい場合に便利。autoMemoryDirectory と同様に影響範囲は限定的だが、ドキュメント管理ワークフローを持つチームには価値がある。

**具体的なユースケース例:**
```json
// .claude/settings.json
{
  "plansDirectory": ".claude/plans"
}
```
- プロジェクトの docs/plans ディレクトリにプランを保存してチームで共有
- タスク管理ツールとの連携のため特定パスにプランファイルを出力

### 14. /config へのリリースチャネル切り替え

**バージョン:** 2.1.3

**機能概要:**
`/config` コマンドから `stable` / `latest` のリリースチャネルを切り替えられるようになった。

**評価:** ★★★☆☆

**理由:**
新機能をいち早く試したい開発者は `latest` に、安定性を重視する本番環境では `stable` を選択するという明確な使い分けが可能になる。エンタープライズ組織がチームごとに異なるチャネルを設定して、段階的なロールアウトを管理できる点も価値がある。

**具体的なユースケース例:**
- 個人の開発環境: `latest` チャネルで最新機能を試用
- チームの共有設定: `stable` チャネルで安定性を確保
- ベータテスター: `latest` で新機能を検証してフィードバックを提供

### 15. /doctor コマンドの Updates セクション追加

**バージョン:** 2.1.6

**機能概要:**
`/doctor` コマンドに Updates セクションを追加。現在のバージョン状態やアップデートの有無を診断結果に含めるようになった。また、到達不能なパーミッションルールの検出・警告機能も追加（2.1.3）。

**具体的なユースケース例:**
```
/doctor   # 環境診断を実行し、Updates・パーミッションルールの問題を確認
```
- 「機能が動かない」場合の最初のデバッグステップとして実行
- パーミッション設定変更後のバリデーションとして使用

### 17. CLAUDE_CODE_ACCOUNT_UUID / CLAUDE_CODE_USER_EMAIL / CLAUDE_CODE_ORGANIZATION_UUID 環境変数

**バージョン:** 2.1.51

**機能概要:**
ユーザーおよび組織の識別情報（アカウントUUID、メールアドレス、組織UUID）を環境変数として参照できるようになった。フックやスクリプトからこれらの情報にアクセス可能。

**評価:** ★★★☆☆

**理由:**
監査ログの生成、マルチテナント環境でのユーザー識別、コスト配賦のためのトラッキングなど、エンタープライズユースケースで重要な識別情報を自動化に組み込める。セキュリティ監査や利用状況のトラッキングに不可欠な基盤となる。

**具体的なユースケース例:**
```bash
# フックスクリプトで利用者を識別して監査ログを記録
echo "$(date): $CLAUDE_CODE_USER_EMAIL used tool $TOOL_NAME" >> /var/log/claude-audit.log

# 組織単位でのコスト配賦レポート
echo "Org: $CLAUDE_CODE_ORGANIZATION_UUID, Tokens: ..." >> /tmp/cost-report.csv
```

### 18. CLAUDE_CODE_DISABLE_CRON 環境変数

**バージョン:** 2.1.72

**機能概要:**
セッション中のスケジュール済みクロンジョブを即時停止できる環境変数。実行中のセッションに対してもクロンを無効化できる。

**評価:** ★★☆☆☆

**理由:**
クロンスケジューリング機能（2.1.71）の導入に伴い、意図しない定期実行を防ぐための安全弁として重要。CI環境や制限された実行環境でクロンが意図せず動作するリスクを排除できる。

**具体的なユースケース例:**
```bash
# CI/CD パイプラインでクロンを無効化して実行
CLAUDE_CODE_DISABLE_CRON=1 claude --print "run tests"
```

### 19. CLAUDE_CODE_TMPDIR 環境変数

**バージョン:** 2.1.5

**機能概要:**
Claude Code が内部で使用する一時ファイルの保存ディレクトリをオーバーライドできる環境変数。

**評価:** ★★☆☆☆

**理由:**
コンテナ環境でのディスク容量管理、セキュリティポリシーによる一時ファイルの保存先制限、特定ストレージへの誘導（高速SSD等）など、インフラ要件がある場合に必要。影響範囲は限定的だが、対象環境では必須の設定になりうる。

**具体的なユースケース例:**
```bash
# /tmp がマウントされていないコンテナ環境で代替パスを指定
export CLAUDE_CODE_TMPDIR=/mnt/fast-ssd/tmp

# tmpfs 上の高速ディレクトリを指定してパフォーマンス改善
export CLAUDE_CODE_TMPDIR=/dev/shm/claude-tmp
```

### 20. CLAUDE_CODE_DISABLE_BACKGROUND_TASKS 環境変数
