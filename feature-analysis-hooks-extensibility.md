# フック・拡張性・スキル・カスタムコマンド 機能分析

Claude Code 新機能一覧（2.0.72以降）から「フック（Hooks）・拡張性・スキル・カスタムコマンド」に関連する機能を抽出し、重要度を評価した結果です。

---

## 採点基準

| 評価 | 基準 |
|------|------|
| ★★★★★ | ゲームチェンジャー。ワークフローを根本的に変える |
| ★★★★☆ | 非常に重要。多くのユーザーに大きな影響 |
| ★★★☆☆ | 重要。特定のユースケースで価値が高い |
| ★★☆☆☆ | 便利。あると嬉しい改善 |
| ★☆☆☆☆ | マイナー。特定の状況でのみ有用 |

---

## 機能一覧と評価

---

### 1. PreToolUse / PostToolUse / Stop フック（エージェント・スキルフロントマターへの拡張）

**バージョン:** 2.1.0

**機能概要:**
エージェント定義およびスキル・スラッシュコマンドのフロントマターで `PreToolUse`、`PostToolUse`、`Stop` フックをサポート。これにより、エージェントやスキル単位でツール使用の前後に任意の処理を挟むことが可能になった。

**評価:** ★★★★★

**評価理由:**
エージェント・スキルにフックを紐付けることで、グローバル設定に頼らずローカルで完結した自律的なワークフロー制御が実現する。たとえば特定のエージェントだけファイル書き込み前に確認スクリプトを走らせる、といった細粒度の制御が可能になり、マルチエージェント構成での安全性と柔軟性が劇的に向上する。ツール制御の単位がセッション全体からエージェント・スキル単位に下がったことは設計思想の転換点といえる。

**ユースケース例:**
```yaml
# .claude/agents/deploy-agent.md フロントマター
---
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "echo '[DEPLOY] About to run: $TOOL_INPUT' >> /tmp/deploy-audit.log"
  Stop:
    - hooks:
        - type: command
          command: "notify-send 'Deploy agent finished'"
---
```

### 2. HTTP フック

**バージョン:** 2.1.63

**機能概要:**
従来のシェルコマンド実行に加え、フックのアクションとして URL に JSON を POST し、レスポンスの JSON を受け取る HTTP フックが追加された。外部サービスやカスタムサーバーとリアルタイムに連携できる。

**評価:** ★★★★★

**評価理由:**
シェルスクリプトを介さずに Webhook・REST API・社内サービスと直接連携できるため、CI/CD パイプラインや監査ログ基盤、承認フローシステムとの統合が飛躍的に容易になる。特に企業環境でのセキュリティ審査・承認プロセスをフック経由で自動化できる点は、エンタープライズ採用の障壁を大きく下げる。

**ユースケース例:**
```json
// settings.json のフック定義例
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "http",
        "url": "https://approval.company.com/api/claude-tool-check",
        "timeout": 5000
      }]
    }]
  }
}
```
シナリオ: 本番環境への bash コマンド実行前に社内の承認サーバーに問い合わせ、未承認なら実行をブロックする。

### 3. InstructionsLoaded フックイベント

**バージョン:** 2.1.69

**機能概要:**
`CLAUDE.md` や `.claude/rules/*.md` がコンテキストに読み込まれた際に発火するフックイベント。セッション初期化の特定タイミングを捕捉し、追加処理を差し込める。

**評価:** ★★★☆☆

**評価理由:**
インストラクションのロード完了を確実なトリガーとして使えるため、プロジェクト固有の環境チェックや補足情報の動的注入が可能になる。ただし用途はやや限定的で、`Setup` フックや `SessionStart` フックで代替できるケースも多い。ルールファイルが複数存在する複雑なプロジェクト構成での活用が主な場面となる。

**ユースケース例:**
```bash
# .claude/settings.json
# InstructionsLoaded 発火時にプロジェクト固有の環境変数チェックを実行
{
  "hooks": {
    "InstructionsLoaded": [{
      "hooks": [{
        "type": "command",
        "command": "scripts/verify-env.sh && echo 'Environment OK'"
      }]
    }]
  }
}
```

### 4. ConfigChange フックイベント

**バージョン:** 2.1.49

**機能概要:**
セッション中に設定ファイルが変更された際に発火するフックイベント。設定変更をリアルタイムに検知して処理を行える。

**評価:** ★★☆☆☆

**評価理由:**
設定変更の検知・通知に使えるが、実用的なユースケースはチーム共有環境での設定同期通知や、設定変更のログ記録といった補助的な用途が中心となる。多くの一般ユーザーにとって頻繁に使う機能ではなく、特定の運用シナリオで価値を発揮する。

**ユースケース例:**
```bash
# 設定変更時にチームの Slack へ通知
{
  "hooks": {
    "ConfigChange": [{
      "hooks": [{
        "type": "http",
        "url": "https://hooks.slack.com/services/xxx/yyy/zzz"
      }]
    }]
  }
}
```

### 5. PostCompact フックイベント

**バージョン:** 2.1.76

**機能概要:**
コンテキストのコンパクション（圧縮）完了後に発火するフックイベント。長時間セッションでの自動コンパクション後に処理を挟める。

**評価:** ★★☆☆☆

**評価理由:**
コンパクション後の状態確認や、圧縮されたコンテキストに対して補足情報を再注入するワークフローで価値を持つ。ただし、コンパクションは自動的に発生するため制御タイミングが限定的であり、長時間セッションを多用するヘビーユーザー向けの機能といえる。

**ユースケース例:**
```bash
# コンパクション後にプロジェクト状態サマリーを再ロード
{
  "hooks": {
    "PostCompact": [{
      "hooks": [{
        "type": "command",
        "command": "cat .claude/project-state.md"
      }]
    }]
  }
}
```

### 6. Setup フックイベント

**バージョン:** 2.1.10

**機能概要:**
`--init`、`--init-only`、`--maintenance` CLI フラグ経由でトリガーされるフックイベント。初期化・メンテナンス処理の自動化に使用する。

**評価:** ★★★☆☆

**評価理由:**
プロジェクトの初期セットアップ処理（依存関係インストール、環境変数チェック、テンプレート生成など）を Claude Code の初期化フローと統合できる。CI 環境や新規メンバーのオンボーディングを自動化する文脈で実用価値が高い。`--init-only` と組み合わせることで完全な環境セットアップスクリプトとして機能する。

**ユースケース例:**
```bash
# .claude/settings.json
{
  "hooks": {
    "Setup": [{
      "hooks": [{
        "type": "command",
        "command": "npm install && cp .env.example .env && echo 'Project initialized'"
      }]
    }]
  }
}

# 実行
claude --init-only
# → npm install と .env コピーが自動実行される
```

### 7. SessionStart フックへの agent_type 追加

**バージョン:** 2.1.2

**機能概要:**
`SessionStart` フック入力に `agent_type` フィールドが追加され、セッション開始時にどのエージェントタイプで起動したかをフック内で識別できるようになった。

**評価:** ★★☆☆☆

**評価理由:**
マルチエージェント構成において、エージェントタイプに応じた初期化処理を分岐させる際に有用。ただし単体では地味な改善であり、複数エージェントを本格的に運用している環境でのみ効果を発揮する。

**ユースケース例:**
```bash
# SessionStart フック内でエージェント種別に応じて異なる初期化を実行
#!/bin/bash
AGENT_TYPE=$(echo "$HOOK_INPUT" | jq -r '.agent_type')
if [ "$AGENT_TYPE" = "deploy-agent" ]; then
  source /etc/deploy-credentials.sh
fi
```

### 8. スキルホットリロード

**バージョン:** 2.1.0

**機能概要:**
`~/.claude/skills` や `.claude/skills` で作成・変更したスキルが、セッションを再起動することなく即座に利用可能になる自動ホットリロード機能。

**評価:** ★★★★☆

**評価理由:**
スキル開発サイクルが劇的に短縮され、試行錯誤しながらスキルを育てる体験が根本的に改善された。従来はスキルを修正するたびにセッションを再起動する必要があったが、ホットリロードにより REPL のような即時フィードバックループが実現する。スキルを積極活用するユーザーにとっては日常的な恩恵を受ける改善。

**ユースケース例:**
```bash
# ターミナル1: スキルを編集中
vim ~/.claude/skills/my-workflow.md

# ターミナル2: Claude Code セッションで即座に反映されたスキルを使用
# /my-workflow と入力すると更新済みスキルが動作する
# セッション再起動不要
```

### 9. スキルフロントマター（context: fork、agent フィールド、allowed-tools YAML）

**バージョン:** 2.1.0

**機能概要:**
スキル・スラッシュコマンドのフロントマターで `context: fork`（フォークされたサブエージェントコンテキストで実行）、`agent` フィールド（使用するエージェントタイプの指定）、`allowed-tools` の YAML リスト形式をサポート。

**評価:** ★★★★☆

**評価理由:**
スキルが単なるプロンプトテンプレートから、エージェント動作の完全な設定単位へと昇格した。`context: fork` により並列処理・独立した作業空間での実行が可能になり、`allowed-tools` でスキルが使えるツールを精密に制御できる。スキルの再利用性・安全性・表現力が一挙に向上するため、スキルベースのワークフロー構築者に大きな恩恵をもたらす。

**ユースケース例:**
```markdown
---
# .claude/skills/safe-review.md フロントマター
context: fork
agent: code-reviewer
allowed-tools:
  - Read
  - Glob
  - Grep
# Bash や Write は使わせない
---
このスキルはコードレビューのみを行い、ファイル変更は一切行いません。
```

### 10. once: true フック設定

**バージョン:** 2.1.0

**機能概要:**
フック定義に `once: true` を設定することで、セッション内で最初の1回だけフックを実行し、以降は無視する動作が可能になった。

**評価:** ★★★☆☆

**評価理由:**
初回ツール使用時の警告表示、一度だけ実行したい環境セットアップ、重複通知の防止など、「最初の1回だけ」というユースケースに対してクリーンな解決策を提供する。フック全体の設計を簡潔に保てる点で、フックを本格活用するユーザーにとって実用度が高い。

**ユースケース例:**
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "once": true,
      "hooks": [{
        "type": "command",
        "command": "echo '警告: このセッションではBashコマンドが実行されます' >&2"
      }]
    }]
  }
}
// → セッション中、最初の Bash 実行時のみ警告を表示。以降は表示しない。
```

### 11. allowed-tools YAML スタイルリスト（フロントマター）

**バージョン:** 2.1.0

**機能概要:**
スキル・スラッシュコマンドのフロントマターで `allowed-tools` フィールドを YAML のリスト形式で記述できるようになった。従来のカンマ区切り文字列よりも可読性が高い。

**評価:** ★★☆☆☆

**評価理由:**
既存の `allowed-tools` 機能の記法改善であり、機能的な新規性はない。ただし複数ツールを指定する際の可読性・保守性が上がり、スキルファイルの品質向上に貢献する。長期的にはスキルの管理コストを下げる効果がある。

**ユースケース例:**
```markdown
---
# 以前の記法
allowed-tools: Read, Glob, Grep

# 新しい YAML リスト記法（どちらも動作する）
allowed-tools:
  - Read
  - Glob
  - Grep
  - Bash(git log *)
---
```

### 12. ${CLAUDE_SKILL_DIR} 変数

**バージョン:** 2.1.69

**機能概要:**
スキルの `SKILL.md` コンテンツ内で `${CLAUDE_SKILL_DIR}` を使用することで、スキルが自身の配置ディレクトリへの絶対パスを参照できるようになった。

**評価:** ★★★☆☆

**評価理由:**
スキルが補助スクリプト・設定ファイル・テンプレートなどのリソースを自身のディレクトリから相対的に参照できるようになり、スキルの自己完結性と移植性が大きく向上する。スキルを単独で配布・共有する場面でハードコードされたパスへの依存を排除できる点は実用上重要。

**ユースケース例:**
```markdown
<!-- .claude/skills/deploy/SKILL.md -->
このスキルを実行するには、以下のスクリプトを使用してください:

```bash
${CLAUDE_SKILL_DIR}/scripts/deploy.sh --env production
```

設定ファイルは `${CLAUDE_SKILL_DIR}/config/deploy.yaml` を参照します。
```

### 13. ${CLAUDE_SESSION_ID} 変数（スキル内）

**バージョン:** 2.1.9

**機能概要:**
スキル内で `${CLAUDE_SESSION_ID}` 文字列置換を使用することで、現在のセッション ID にアクセスできるようになった。セッションをまたいだ状態管理やログの紐付けに活用できる。

**評価:** ★★★☆☆

**評価理由:**
複数の並列セッションや長期間にわたる作業ログの管理において、セッション ID を一意な識別子として活用できる。スキルが生成した成果物（ログ、中間ファイルなど）をセッション単位で整理するユースケースで特に有効。マルチセッション運用を行う上級ユーザーにとって実用価値が高い。

**ユースケース例:**
```markdown
<!-- .claude/skills/audit-logger/SKILL.md -->
作業ログを以下のファイルに記録してください:

/tmp/claude-audit/${CLAUDE_SESSION_ID}/actions.log

各操作の前後にタイムスタンプと操作内容を記録し、
セッション終了時にサマリーを生成してください。
```

### 14. カスタムコマンド引数（$0, $1 等のショートハンド）

**バージョン:** 2.1.19

**機能概要:**
カスタムコマンド（スラッシュコマンド）の実行時に渡された引数を `$0`、`$1` などのショートハンドで個別にアクセスできるようになった。

**評価:** ★★★★☆

**評価理由:**
カスタムコマンドがパラメータ化された本格的なツールとして機能するようになる転換点となる機能。引数を受け取れることで、汎用的なワークフロー自動化（「このブランチに対してレビューを実行」「この PR 番号のテストを走らせる」など）をコマンド一本で表現できる。プログラマブルなスラッシュコマンドエコシステムの基盤となる。

**ユースケース例:**
```markdown
<!-- .claude/commands/pr-review.md -->
GitHub PR #$1 のコードレビューを実施してください。

1. `gh pr diff $1` でdiffを取得
2. 変更点を分析して問題点を指摘
3. `gh pr review $1 --comment -b "..."` でレビューを投稿

レビュアー: $2（省略時は "Claude"）
```
```bash
# 使用例
/pr-review 1234 hirakawa
# → PR #1234 を hirakawa としてレビュー
```

### 15. PreToolUse フックの additionalContext 返却

**バージョン:** 2.1.9

**機能概要:**
`PreToolUse` フックがモデルへ `additionalContext` を返却できるようになった。ツール実行前にフックから動的なコンテキスト情報をモデルに注入できる。

**評価:** ★★★★★

**評価理由:**
フックが単なる「監視・ブロック」の役割を超え、モデルの判断に積極的に情報を提供できる能動的な存在になった。たとえば、ファイル編集前に関連する設計ドキュメントを自動注入したり、bash 実行前に環境情報を動的に追加したりすることで、モデルの作業品質を外部から高められる。エージェントループを拡張する最も強力な仕組みの一つ。

**ユースケース例:**
```bash
#!/bin/bash
# PreToolUse フックスクリプト: Bash 実行前に実行環境情報を注入
TOOL=$(echo "$HOOK_INPUT" | jq -r '.tool_name')
if [ "$TOOL" = "Bash" ]; then
  CONTEXT=$(cat <<EOF
{
  "additionalContext": "現在の環境: $(uname -a)\nPATH: $PATH\nGit branch: $(git branch --show-current 2>/dev/null)\n本番環境への直接デプロイは禁止されています。"
}
EOF
)
  echo "$CONTEXT"
fi
```

### 16. fook イベントへの agent_id / agent_type フィールド追加

**バージョン:** 2.1.69

**機能概要:**
フックイベントの入力データに `agent_id`（サブエージェント識別子）と `agent_type`（エージェント種別）フィールドが追加され、どのエージェントから発火したフックかを識別できるようになった。

**評価:** ★★★☆☆

**評価理由:**
マルチエージェント環境でのフック処理を精密化できる。どのエージェントが何をしようとしているかを識別して、エージェント別の権限制御やログ分離が実現する。マルチエージェントを本格運用している組織にとっては重要だが、単一エージェント利用者には影響しない。

**ユースケース例:**
```bash
#!/bin/bash
# フック内でエージェント別に権限チェック
AGENT_TYPE=$(echo "$HOOK_INPUT" | jq -r '.agent_type')
TOOL=$(echo "$HOOK_INPUT" | jq -r '.tool_name')

if [ "$AGENT_TYPE" = "readonly-researcher" ] && [ "$TOOL" = "Write" ]; then
  echo '{"decision": "block", "reason": "リサーチエージェントはファイル書き込み禁止"}'
  exit 0
fi
```

### 17. WorktreeCreate / WorktreeRemove フックイベント

**バージョン:** 2.1.50

**機能概要:**
git worktree の作成・削除時に発火するフックイベント。worktree のライフサイクルに連動した処理を自動化できる。

**評価:** ★★☆☆☆

**評価理由:**
`--worktree` フラグを活用した並列開発ワークフローにおいて、worktree の初期化（依存関係インストール、環境ファイルコピー）や後片付け（一時ファイル削除、通知送信）を自動化できる。ただし worktree 機能自体がやや上級者向けであり、対象ユーザー層が限定される。

**ユースケース例:**
```json
{
  "hooks": {
    "WorktreeCreate": [{
      "hooks": [{
        "type": "command",
        "command": "cd $WORKTREE_PATH && cp ../.env . && npm install"
      }]
    }],
    "WorktreeRemove": [{
      "hooks": [{
        "type": "command",
        "command": "echo 'Worktree $WORKTREE_NAME removed' | notify"
      }]
    }]
  }
}
```

### 18. TeammateIdle / TaskCompleted フックイベント

**バージョン:** 2.1.33

**機能概要:**
マルチエージェントワークフロー向けに、チームメイトエージェントがアイドル状態になった時（`TeammateIdle`）とタスクが完了した時（`TaskCompleted`）に発火するフックイベントを追加。

**評価:** ★★★☆☆

**評価理由:**
エージェントチーム機能と組み合わせることで、エージェント間の協調処理（あるエージェントが完了したら次のエージェントに通知する、全員がアイドルになったら集計処理を走らせるなど）をフックで自動化できる。エージェントチーム機能が正式リリースされた際には重要度がさらに上がる見込み。

**ユースケース例:**
```json
{
  "hooks": {
    "TaskCompleted": [{
      "hooks": [{
        "type": "http",
        "url": "https://ci.company.com/api/agent-task-done",
        "method": "POST"
      }]
    }]
  }
}
// → タスク完了時に CI システムへ通知し、次のパイプラインステップをトリガー
```

### 19. Elicitation / ElicitationResult フック

**バージョン:** 2.1.76

**機能概要:**
MCP サーバーが構造化入力をリクエストする際（Elicitation）に発火するフック。レスポンスをインターセプトして自動入力やオーバーライドが可能。

**評価:** ★★★☆☆

**評価理由:**
MCP サーバーとのインタラクティブなフォーム入力を自動化・カスタマイズできる。CI/CD 環境での無人実行や、組織ポリシーに基づく入力値の強制適用に活用できる。MCP を本格活用している組織では重要だが、一般ユーザーには直接関係しないケースが多い。

**ユースケース例:**
```bash
#!/bin/bash
# ElicitationResult フック: MCP からの入力要求に自動応答
INPUT=$(echo "$HOOK_INPUT" | jq -r '.elicitation.field_name')
if [ "$INPUT" = "environment" ]; then
  echo '{"override": {"environment": "staging"}}'
fi
# → MCP が環境選択を求めてきたら常に "staging" を自動選択
```

### 20. /reload-plugins コマンド

**バージョン:** 2.1.69

**機能概要:**
`/reload-plugins` スラッシュコマンドでセッションを再起動せずにプラグイン変更を即座に反映できる。

**評価:** ★★★☆☆

**評価理由:**
スキルホットリロード（2.1.0）と同様の思想で、プラグイン開発サイクルを高速化する。プラグインを自作・カスタマイズして運用している開発者にとっては日常的に恩恵を受ける機能。ただしホットリロード対応の範囲が限定的なユーザーには無関係。

**ユースケース例:**
```bash
# プラグインを修正した後
vim ~/.claude/plugins/my-plugin/index.js

# Claude Code セッションに戻り、再起動不要でリロード
/reload-plugins
# → 修正が即座に反映される
```

## 注目すべきトレンド

1. **フックの能動化**: 2.1.9 の `additionalContext` 返却により、フックは「監視・ブロック」から「モデルへの情報注入」へと役割が拡大した。これはエージェントループの制御パラダイムを大きく変える。

2. **スキルの自律化**: フロントマター、`context: fork`、`${CLAUDE_SKILL_DIR}` などが揃うことで、スキルが自己完結した独立したエージェント的存在として機能するようになった。

3. **外部統合の容易化**: HTTP フックにより、シェルスクリプトを介さずに企業の既存システム（CI/CD、承認フロー、Slack、監査ログ）と直接統合できるようになった。エンタープライズ採用の促進につながる重要な変化。

4. **マルチエージェント対応のフック強化**: `agent_id`、`agent_type`、`TeammateIdle`、`TaskCompleted` など、マルチエージェント環境に特化したフックイベントが継続的に追加されており、今後のエージェントチーム機能の本格化を見据えた準備が進んでいる。
