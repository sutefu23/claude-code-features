# Claude Code 新機能分析：MCP・外部連携・統合

対象バージョン範囲: 2.0.73 〜 2.1.76
抽出カテゴリ: MCP関連・OAuth・外部サービス連携・プラグイン・マーケットプレイス

---

## 機能一覧と採点

---

### 1. MCP elicitation サポート（v2.1.76）

**機能概要:**
MCPサーバーがタスク実行中に構造化入力をインタラクティブダイアログ（フォームフィールドやブラウザURL）でユーザーに要求できるようになった。あわせて `Elicitation` / `ElicitationResult` フックを追加し、レスポンス送信前にインターセプト・オーバーライドが可能。

**採点: ★★★★★**

**評価理由:**
MCPサーバーが一方向の情報提供にとどまらず、双方向のインタラクティブな入力収集が可能になったことで、MCPサーバーの設計の幅が根本的に広がる。認証フロー・パラメータ収集・確認ステップなど、これまでハードコードや環境変数頼みだった部分をUI駆動で実装できる。フックによるインターセプトも加わり、エンタープライズ環境での承認ワークフローへの応用が可能。

**具体的なユースケース例:**
- デプロイ先環境（staging/production）をダイアログで選択させるMCPサーバーを構築し、誤操作を防止する
- データベース接続情報をセッション開始時にフォームで入力させ、平文ハードコードを排除する
- `Elicitation` フックで入力値をバリデーションし、不正な値をオーバーライドして安全なデフォルト値に差し替える

---

### 2. MCP `list_changed` 通知サポート（v2.1.0）

**機能概要:**
再接続なしでMCPサーバーのツール・プロンプト・リソースを動的に更新可能になった。MCPサーバー側が `list_changed` 通知を送ることでクライアント側が自動的にツール一覧を再取得する。

**採点: ★★★★★**

**評価理由:**
従来はMCPサーバーのツール構成が変わるたびにセッションの再接続が必要だったが、これが不要になりセッションの継続性が大幅に改善される。長時間のエージェントワークフローや、動的にツールを追加・削除するMCPサーバー（権限に応じてツールを出し分けるなど）との相性が格段に向上する。

**具体的なユースケース例:**
- ユーザーが追加のAPI権限を取得した際、MCPサーバーが新ツールをホットリロードし、セッション継続のまま即座に利用開始できる
- A/Bテスト用のMCPサーバーが実行中にツールセットを切り替えるシナリオ
- 開発中のMCPサーバーのツールを追加・修正しながら、セッションを再起動せずに動作確認する

---

### 3. claude.ai MCPコネクターをClaude Codeで使用可能に（v2.1.46）

**機能概要:**
claude.aiで設定・管理しているMCPコネクター（サーバー接続）をClaude Codeからそのまま利用できるようになった。

**採点: ★★★★★**

**評価理由:**
WebUIとCLIツール間でMCP設定を二重管理する手間がなくなり、ユーザー体験が統一される。claude.aiで一度設定したMCPサーバーをClaude Codeでも即座に使えることで、エコシステム全体の一貫性が確立される。特にチーム環境ではMCP設定の標準化・共有が容易になる。

**具体的なユースケース例:**
- claude.aiでSlack MCPコネクターを設定し、Claude Codeでコーディング中にも同じSlack連携を利用してチャンネルへの投稿や情報取得を行う
- Webブラウザで設定済みのDB接続MCPを、Claude CodeのCLIセッションでもそのまま活用する
- `ENABLE_CLAUDEAI_MCP_SERVERS=false` で特定の環境（CIなど）では無効化し、用途に応じて制御する

---

### 4. MCP OAuth認証：手動URLペーストフォールバック（v2.1.63）

**機能概要:**
MCP OAuth認証フローで、ブラウザが自動で開かない環境（SSH・ヘッドレスサーバーなど）でも、URLを手動でコピー＆ペーストして認証を完了できるフォールバック手段を追加。

**採点: ★★★☆☆**

**評価理由:**
リモートサーバーやCI環境など、ブラウザが使えない状況でのMCP OAuth認証の障壁を取り除く実用的な改善。既存機能の穴を埋めるものでありゲームチェンジャーではないが、OAuth対応MCPを本番環境で運用する際の必須要件を満たす。

**具体的なユースケース例:**
- SSHでリモートサーバーにClaude Codeをインストールし、GitHub MCPサーバーのOAuth認証をURLコピーで完了させる
- Dockerコンテナ内のClaude Codeで、ホストのブラウザに認証URLを貼り付けてGoogle Drive MCPの認証を行う

---

### 5. MCP OAuth：事前設定済みクライアントクレデンシャルのサポート（v2.1.30）

**機能概要:**
MCPサーバーの起動時に `--client-id` / `--client-secret` フラグで事前にOAuthクライアントクレデンシャルを設定できるようになった。

**採点: ★★★★☆**

**評価理由:**
エンタープライズ環境で組織管理のOAuthアプリを使ってMCPサーバーを認証する際に、インタラクティブな認証フローを省略してCI/CD統合や自動化が可能になる。個人ユーザーよりもチーム・企業での大規模運用において、管理コストを大幅に削減できる重要機能。

**具体的なユースケース例:**
```bash
# GitHubのOAuth App認証情報をMCPサーバー起動時に事前設定
claude --mcp-server "github --client-id $GH_CLIENT_ID --client-secret $GH_CLIENT_SECRET"
```
- エンタープライズで管理する専用OAuthアプリを全メンバーのMCP設定に配布し、個別認証を不要にする
- CI環境でMCP認証を非インタラクティブに完了させ、自動化パイプラインに組み込む

---

### 6. `oauth.authServerMetadataUrl` 設定（v2.1.69）

**機能概要:**
MCPサーバーのカスタムOAuthメタデータ検出URLを設定ファイルで指定可能になった。標準のディスカバリーエンドポイント以外のOAuth認証サーバーを利用できる。

**採点: ★★★☆☆**

**評価理由:**
社内認証基盤（Keycloak・Okta・Active Directoryなど）をMCP OAuth認証に使いたいエンタープライズ環境での導入障壁を下げる機能。汎用性は高くないが、該当環境では必須となりうる重要なカスタマイズポイント。

**具体的なユースケース例:**
- Keycloakを社内OAuthプロバイダーとして使っている組織で、`oauth.authServerMetadataUrl` に社内Keycloakのメタデータエンドポイントを設定し、MCP認証を社内SSO経由で行う
- 独自OAuthサーバーを持つ企業が、標準ではない検出URLを持つプロバイダーとMCPを連携させる

---

### 7. Team プラン OAuthユーザーへのポリシー制限取得を拡大（v2.1.69）

**機能概要:**
リモートコントロール制限等のポリシー制限の取得対象を、これまでのEnterpriseプランに加えTeamプランのOAuthユーザーにも拡大した。

**採点: ★★★☆☆**

**評価理由:**
Teamプランのユーザーもエンタープライズと同等のポリシー制御を受けられるようになり、組織全体でのセキュリティ・コンプライアンス管理が一貫できる。Teamプランを使う中規模チームのセキュリティ運用にとって実質的な価値がある。

**具体的なユースケース例:**
- Teamプランの組織で管理者が `remoteControlRestrictions` を設定し、OAuthログインしているメンバー全員に適用させる
- TeamプランのCI環境でOAuth認証を使いながら、管理者設定のポリシーに準拠した操作制限を自動的に受ける

---

### 8. `ENABLE_CLAUDEAI_MCP_SERVERS=false` 環境変数（v2.1.63）

**機能概要:**
claude.ai MCPサーバーの利用を環境変数でオプトアウトできるようになった。

**採点: ★★☆☆☆**

**評価理由:**
CI/CDやセキュリティ要件の厳しい環境でclaude.ai MCPサーバーへの接続を明示的に無効化できる。機能追加というよりセーフガードとしての側面が強く、MCP統合を使う環境の制御性を高める補完的な改善。

**具体的なユースケース例:**
```bash
# CIパイプラインでclaude.ai MCPサーバーを無効化して実行
ENABLE_CLAUDEAI_MCP_SERVERS=false claude --print "run tests"
```
- ネットワーク制限のある企業環境でclaude.ai外部通信を遮断してローカルMCPのみ使用する

### 9. `auto:N` 構文でMCPツール検索の自動有効化しきい値を設定（v2.1.9）

**機能概要:**
`auto:N`（Nはコンテキストウィンドウの割合 0-100）という構文で、MCPのツール検索機能を自動的に有効にするしきい値を設定できるようになった。

**採点: ★★★☆☆**

**評価理由:**
多数のMCPツールを登録している環境でコンテキスト効率を最適化するための重要なチューニングポイント。手動管理なしにコンテキスト状況に応じてMCPツールの検索戦略を動的に切り替えられるため、大規模MCPセットアップの実用性が高まる。

**具体的なユースケース例:**
- 50個以上のMCPツールを登録している環境で `auto:30` を設定し、コンテキストウィンドウの30%を超えたら自動的にツール検索を無効化してコンテキストを節約する
- `auto:0` でデフォルトの自動有効化を無効にし、`auto:100` で常に有効にする

---

### 10. プラグインをnpmからインストール時のカスタムレジストリと特定バージョンピニングをサポート（v2.1.51）

**機能概要:**
npmソースからプラグインをインストールする際に、プライベートnpmレジストリの指定と特定バージョンへのピン留めができるようになった。

**採点: ★★★★☆**

**評価理由:**
エンタープライズ環境でプライベートnpmレジストリ（Artifactory・GitHub Packages・npm Enterprise等）にホストした社内プラグインを安全に配布・バージョン管理できるようになる。再現性のある環境構築とセキュリティ審査済みプラグインの強制利用を両立でき、組織規模での運用に不可欠な機能。

**具体的なユースケース例:**
- 社内プライベートnpmレジストリ（`https://npm.company.internal`）からセキュリティ審査済みの社内プラグインをインストールする
- プラグインを `@my-org/claude-plugin@1.2.3` のように特定バージョンにピン留めし、全開発者が同一バージョンを使用するよう強制する
- CIパイプラインでバージョンピニングにより再現性を確保する

---

### 11. `.git` サフィックスなしのマーケットプレイスgit URLをサポート（v2.1.72）

**機能概要:**
Azure DevOpsやAWS CodeCommitなど、URLに `.git` サフィックスが付かないGitホスティングサービスのURLをマーケットプレイスのgitソースとして指定できるようになった。

**採点: ★★★☆☆**

**評価理由:**
Azure DevOpsやAWS CodeCommitを利用している企業でも、マーケットプレイス経由のプラグイン・スキル配布が正常に機能するようになる。対象ユーザーは限定されるが、これらのプラットフォームを標準とする組織では従来できなかったワークフローが実現できる。

**具体的なユースケース例:**
- Azure DevOpsリポジトリ（`https://dev.azure.com/org/project/_git/my-plugin`）をマーケットプレイスソースに追加し、社内プラグインをチームに配布する
- AWS CodeCommit上の社内ツールリポジトリをClaudeマーケットプレイスに登録して管理する

---

### 12. プラグインソースタイプ `git-subdir` のサポート（v2.1.69）

**機能概要:**
gitリポジトリ全体ではなく、リポジトリ内の特定サブディレクトリをプラグインソースとして指定できるようになった。

**採点: ★★★☆☆**

**評価理由:**
モノレポ構成でプラグインをツールやライブラリと同じリポジトリで管理している組織にとって、プラグイン配布のために別リポジトリを作る必要がなくなる。既存のリポジトリ構成を変えずにプラグインのソース管理が可能になり、インフラ管理の複雑さを軽減できる。

**具体的なユースケース例:**
- `monorepo/tools/claude-plugins/` にプラグインが置かれたモノレポから、そのサブディレクトリだけをプラグインソースとして登録する
- 既存の社内ツールリポジトリに `claude/plugins/` ディレクトリを追加し、リポジトリ分割なしにプラグインを管理・配布する

---

### 13. `strictKnownMarketplaces` に `pathPattern` を追加（v2.1.69）

**機能概要:**
マーケットプレイスのソース検証で、ファイル・ディレクトリソースに対して正規表現パターンマッチングによるパス制限が可能になった。

**採点: ★★☆☆☆**

**評価理由:**
特定ディレクトリ配下のみを許可するなど、マーケットプレイスソースのセキュリティポリシーをより細かく制御できる。セキュリティ要件の高いエンタープライズ環境での制御強化に役立つが、一般ユーザーには影響しない管理者向け機能。

**具体的なユースケース例:**
- `pathPattern: "^/approved-plugins/"` を設定し、組織が承認したディレクトリ配下のプラグインのみインストールを許可する
- 特定のgitリポジトリパターンに一致するソースのみを信頼済みマーケットプレイスとして登録する

---

### 14. `pluginTrustMessage` でプラグインインストール時の信頼警告に組織固有コンテキストを付加（v2.1.69）

**機能概要:**
プラグインインストール時に表示される信頼警告ダイアログに、組織固有のメッセージやガイダンスを追加できるようになった。

**採点: ★★☆☆☆**

**評価理由:**
エンタープライズ管理者がプラグインのセキュリティポリシーや承認プロセスをユーザーに案内するカスタムメッセージを表示でき、組織のコンプライアンス対応に貢献する。影響範囲は限定的だが、大規模組織での運用管理の質を高める実用的な機能。

**具体的なユースケース例:**
- `pluginTrustMessage: "プラグインのインストールは社内セキュリティポリシーに従ってください。承認済みリストは社内ポータルを参照。"` と設定して全メンバーへガイダンスを表示する

---

### 15. `/reload-plugins` コマンド（v2.1.69）

**機能概要:**
セッションを再起動せずにプラグインの変更をリロードできるコマンドを追加。プラグイン開発中の反復サイクルを短縮できる。

**採点: ★★★☆☆**

**評価理由:**
プラグインを開発・カスタマイズしているユーザーにとって、セッション再起動なしで変更を即反映できるのは開発効率を大きく向上させる。v2.1.0のスキルホットリロードと同様の考え方をプラグインにも適用したもので、開発反復速度が実質的に向上する。

**具体的なユースケース例:**
- プラグインのスクリプトを編集後、`/reload-plugins` を実行して即座に動作確認する（セッション履歴・コンテキストを保持したまま）
- CI経由でプラグインを自動デプロイした後、`/reload-plugins` でセッション断なく新バージョンを適用する

---

### 16. `--add-dir` で指定ディレクトリからの `enabledPlugins` と `extraKnownMarketplaces` 読み込み（v2.1.45）

**機能概要:**
`--add-dir` で追加したディレクトリに置かれた設定ファイルから、有効化プラグインリストや追加マーケットプレイス定義を読み込めるようになった。

**採点: ★★★☆☆**

**評価理由:**
プロジェクトリポジトリにプラグイン設定を含めることで、リポジトリをクローンするだけでチーム全員が同じプラグイン環境を自動的に取得できる。プロジェクトごとのプラグイン構成のバージョン管理が可能になり、チーム間の環境差異を解消できる。

**具体的なユースケース例:**
- `.claude/settings.json` に `enabledPlugins` を定義したリポジトリを `--add-dir .` で追加し、チームメンバー全員が自動的に必要なプラグインを有効化した状態で開発を開始できる

---

### 17. SSL証明書エラー時のガイダンス追加（v2.1.73）

**機能概要:**
OAuthログインや接続チェックでSSL証明書エラー（企業プロキシや `NODE_EXTRA_CA_CERTS` 起因）が発生した場合に、解決策を示すガイダンスメッセージを表示するようになった。

**採点: ★★☆☆☆**

**評価理由:**
企業プロキシ環境はClaude Codeの導入障壁として頻出の問題であり、適切なガイダンスでサポートコストを削減できる。機能追加というよりUX改善だが、エンタープライズ導入のスムーズさに直結するため実際の価値は高い。

**具体的なユースケース例:**
- 社内MITM型プロキシ環境でOAuthログインがSSLエラーで失敗した際に、`NODE_EXTRA_CA_CERTS=/path/to/corp-ca.crt` を設定するよう案内を受ける

---

### 18. `sandbox.enableWeakerNetworkIsolation` 設定（v2.1.69）

**機能概要:**
macOSのサンドボックス環境でMITMプロキシを使用時に、`gh`・`gcloud`・`terraform` 等のツールのTLS証明書検証を許可する設定を追加。

**採点: ★★☆☆☆**

**評価理由:**
企業プロキシ環境での外部CLIツールとの連携問題を解消する実用的な設定だが、「弱いネットワーク分離」という名の通りセキュリティとのトレードオフを伴い、慎重な運用が必要。対象ユーザーは限定的だが該当環境では作業継続に不可欠。

**具体的なユースケース例:**
- 社内MITMプロキシを経由して `gh` コマンドや `gcloud` CLIを使う必要がある企業環境で、TLSエラーを回避してGitHub・GCP操作を可能にする

---

### 19. `claude auth login` / `claude auth status` / `claude auth logout` CLIサブコマンド（v2.1.41）

**機能概要:**
OAuth認証の操作をCLIサブコマンドとして実行できるようになった。従来のインタラクティブなログインプロセスをスクリプト化・自動化しやすくなった。

**採点: ★★★★☆**

**評価理由:**
CI/CDパイプラインやDocker環境・スクリプトからClaudeの認証状態を管理できるようになる。特に `claude auth status` でスクリプトから認証状態を確認できることは、自動化ワークフロー構築において重要な基盤機能となる。

**具体的なユースケース例:**
```bash
# CI環境での認証チェックと自動ログイン
claude auth status || claude auth login --api-key "$CLAUDE_API_KEY"

# Dockerfileでの認証セットアップ
RUN claude auth login --api-key "${CLAUDE_API_KEY}"
```
- Gitフックの中で `claude auth status` を呼び出し、認証済みの場合のみClaude Codeによるコードレビューを実行する

---

### 20. `claude remote-control` サブコマンドを全ユーザーに提供（v2.1.51）

**機能概要:**
外部ビルドツール・IDE・スクリプトからClaude Codeをローカルで制御するためのリモートコントロール機能を全ユーザーに開放した。

**採点: ★★★★☆**

**評価理由:**
Claude Codeを外部ツールから制御できることで、VS Code拡張・JetBrains Plugin・カスタムIDEなどとの統合が可能になる。開発ツールチェーンにClaude Codeを組み込む際の核心的なAPIとなり、エコシステムの拡張性を大きく高める。

**具体的なユースケース例:**
- VS Code拡張からローカルのClaude Codeセッションにコマンドを送信し、エディタ内で選択したコードのリファクタリングを指示する
- `claude remote-control --name my-session` で名前付きセッションを起動し、外部スクリプトから特定セッションを制御する
