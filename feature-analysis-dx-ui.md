# 開発者体験（DX）・UI・キーバインド・Vim・ターミナル 機能分析

対象: Claude Code 2.0.73 〜 2.1.76 の新機能から抽出
分析日: 2026-03-14

---

## 採点基準

- ★★★★★: ゲームチェンジャー。ワークフローを根本的に変える
- ★★★★☆: 非常に重要。多くのユーザーに大きな影響
- ★★★☆☆: 重要。特定のユースケースで価値が高い
- ★★☆☆☆: 便利。あると嬉しい改善
- ★☆☆☆☆: マイナー。特定の状況でのみ有用

---

## 機能一覧と採点

---

### 1. カスタマイズ可能なキーボードショートカット

**バージョン:** 2.1.18

**機能の概要:**
コンテキスト別のキーバインディング、コードシーケンス、ワークフローのパーソナライズが可能。`/keybindings` コマンドで設定を開始できる。

**採点:** ★★★★★

**評価理由:**
Claude Code を毎日使うユーザーにとって、キーバインドのカスタマイズは繰り返し操作の効率を劇的に向上させる。既存のエディタ（Vim/Emacs/VS Code）のキーバインドに合わせることで、コンテキストスイッチのストレスが消える。一度設定すれば全セッションで恩恵を受け続けるため、投資対効果が極めて高い。

**具体的なユースケース例:**
- `/keybindings` を実行し、`chat:submit` を `Ctrl+J`（Enterの代替）にバインド
- `voice:pushToTalk` を `Ctrl+Space` に変更して音声入力と IME 切り替えのバッティングを回避
- `chat:newline` を `Shift+Enter` に設定してマルチライン入力を快適化

---

### 2. 新しい Vim モーション追加

**バージョン:** 2.1.0

**機能の概要:**
Vim ノーマルモードで新たなモーションを追加。`;`、`,`、`y`（ヤンク）、`yy`/`Y`、`p`/`P`（ペースト）、テキストオブジェクト（`iw`、`aw` 等）、インデント操作（`>>`、`<<`）、行結合（`J`）をサポート。

**採点:** ★★★★☆

**評価理由:**
Vim ユーザーにとってテキストオブジェクトとヤンク/ペーストは Vim 操作の核心であり、これらなしでは「Vim ライク」と呼べない。`iw`/`aw` などのテキストオブジェクトが使えることで、単語・文節単位の編集が直感的になる。Vim ユーザー層には非常に大きなインパクトだが、非 Vim ユーザーには無関係のため ★★★★☆。

**具体的なユースケース例:**
- プロンプト内の単語を素早く置換: `ciw` で単語を削除して入力モードへ
- コードスニペットをヤンク: `yy` で行をコピーし `p` で貼り付け
- インデント調整: `>>` でブロックをインデント、`<<` で戻す
- `f` で文字を検索後、`;` で次の一致に素早くジャンプ

---

### 3. Vim ノーマルモードでの矢印キーによるヒストリナビゲーション

**バージョン:** 2.1.20

**機能の概要:**
Vim ノーマルモードでカーソル移動が不可な位置にいる場合、矢印キーでコマンド入力履歴のナビゲーションが可能になった。

**採点:** ★★☆☆☆

**評価理由:**
Vim モード利用者向けの細かな使い勝手改善。Vim ユーザーは通常 `k`/`j` でヒストリ操作することに慣れているが、特定の状況で矢印キーが使えることで戸惑いを減らす。既存ワークフローを根本的に変えるものではないが、Vim モード利用者の不満を解消するパッチ的な改善。

**具体的なユースケース例:**
- ノーマルモードで入力の先頭にいる際、上矢印キーで前回のプロンプトを呼び出す
- 長いプロンプト履歴を矢印キーで素早くブラウズし、直前のコマンドを再利用する

---

### 4. `/terminal-setup` で Kitty、Alacritty、Zed、Warp をサポート

**バージョン:** 2.0.74

**機能の概要:**
`/terminal-setup` コマンドで Kitty、Alacritty、Zed、Warp の各ターミナルエミュレータ向けの設定ガイドと自動セットアップを提供。

**採点:** ★★★☆☆

**評価理由:**
Claude Code の初期設定のハードルを大幅に下げる機能。特に Kitty や Warp のような高機能ターミナルでは手動設定が煩雑で、初心者がつまずく原因になっていた。モダンターミナルを使う開発者にとって「動かない」という初期体験の悪さを防ぐ重要なオンボーディング改善。

**具体的なユースケース例:**
- Kitty ユーザーが `/terminal-setup` を実行してフォント・キーバインドの互換性を自動設定
- Warp ユーザーが Claude Code と Warp の AI 機能が競合しないよう設定を最適化
- 新メンバーが Mac に Claude Code をインストール後、即座に `/terminal-setup` でセットアップ完了

---

### 5. `/theme` で構文ハイライトのオン/オフ切り替え（Ctrl+T）

**バージョン:** 2.0.74

**機能の概要:**
`/theme` コマンドでテーマピッカーを開き、`Ctrl+T` で構文ハイライトのオン/オフをトグル可能。テーマピッカーに構文ハイライト情報も追加。

**採点:** ★★☆☆☆

**評価理由:**
視認性の好みは人それぞれであり、ハイライトが逆に邪魔になるケース（特定のカラースキームやアクセシビリティ要件）も存在する。日常的に切り替える機能ではないが、一度好みのテーマを設定できれば快適度が上がる。コア機能への影響は小さいため ★★☆☆☆。

**具体的なユースケース例:**
- ダークターミナル環境で `/theme` を開き、`Ctrl+T` でハイライトオフにして視認性を改善
- プレゼンテーション・スクリーンキャスト時に高コントラストテーマに切り替え
- バッテリー節約やターミナルパフォーマンスを優先する場合にハイライトを無効化

---

### 6. `/color` コマンドでプロンプトバーの色を設定

**バージョン:** 2.1.75

**機能の概要:**
`/color` コマンドでプロンプトバーのアクセントカラーを自由に設定可能。

**採点:** ★★☆☆☆

**評価理由:**
複数のプロジェクトや環境（本番/ステージング等）を同時に扱う際、色で視覚的に区別できるのは実用的なメリット。しかし純粋な美観のカスタマイズとしての側面も強く、ワークフローへの影響は限定的。マルチセッション環境での誤操作防止には有効。

**具体的なユースケース例:**
- 本番環境用の Claude Code セッションを赤系、開発環境を青系に設定して誤操作防止
- チームで色コーディングのルールを決め、プロジェクトごとに異なる色を割り当て
- `/color #FF6B35` のように直接カラーコードを指定してブランドカラーに合わせる

---

### 7. `/copy` コマンドでコードブロックのインタラクティブピッカー

**バージョン:** 2.1.59

**機能の概要:**
`/copy` 実行時にコードブロックが含まれる場合、インタラクティブなピッカーを表示し、個別のコードブロックまたはレスポンス全体を選択してクリップボードにコピー可能。

**採点:** ★★★★☆

**評価理由:**
Claude のレスポンスから特定のコードブロックだけをコピーしたいシーンは頻繁に発生する。従来はマウスで手動選択するか全体をコピーして不要部分を削除する必要があったが、このピッカーにより一発で目的のコードだけを取得できる。ターミナル環境での作業効率を大きく改善するため ★★★★☆。

**具体的なユースケース例:**
- Claude が複数の代替実装を提案した際、特定のコードブロックだけを選択してコピー
- ドキュメントとコードが混在したレスポンスから、コードのみを抽出してファイルに貼り付け
- SSH 環境で複数のコマンドスニペットが返ってきた際、必要なものだけを選んでコピー

---

### 8. `/copy` で `w` キーを押してファイルに直接書き出し

**バージョン:** 2.1.72

**機能の概要:**
`/copy` ピッカーで `w` キーを押すと、選択した内容をクリップボード経由ではなくファイルに直接書き出し可能。SSH 環境での利用に特に有効。

**採点:** ★★★☆☆

**評価理由:**
SSH 環境ではクリップボードが使えないため、コードを手元に持ち帰る手段が限られていた。ファイルへの直接書き出しはこの問題を解決し、リモート開発者の痛点を解消する。SSH を使わない開発者には恩恵が薄いが、リモートサーバーで Claude Code を使うユーザーにとってはゲームチェンジャー的な改善。

**具体的なユースケース例:**
- SSH 接続したサーバーで Claude が生成したスクリプトを `/copy` → `w` → ファイル名指定で即座に保存
- クリップボードが使えない CI/CD 環境で生成コードをファイルとして出力
- `/copy` → `w` → `~/output.py` でコード生成結果をすぐにスクリプトとして実行可能な状態に

---

### 9. `/copy` に「Always copy full response」オプション追加

**バージョン:** 2.1.63

**機能の概要:**
`/copy` ピッカーに「常にレスポンス全体をコピー」オプションを追加。毎回ピッカーで選択する手間を省ける。

**採点:** ★★☆☆☆

**評価理由:**
2.1.59 で追加されたインタラクティブピッカーの実用性を高める補完的な機能。全体コピーが多い場合はピッカーを毎回操作する手間を省ける。単独ではマイナーな改善だが、`/copy` ワークフロー全体の完成度を高める。

**具体的なユースケース例:**
- ドキュメント生成用途で常にレスポンス全体が必要なワークフローで「Always copy」を選択
- Markdown や JSON 全体をコピーして外部ツールに貼り付ける繰り返し作業の高速化

---

### 10. 外部エディタ起動ショートカット（Ctrl+G）のヘルプメニュー表示

**バージョン:** 2.1.20

**機能の概要:**
ヘルプメニューに外部エディタ（`$EDITOR` 設定に基づく）を起動するショートカット Ctrl+G を追加。長いプロンプトや複雑な指示の編集に活用可能。

**採点:** ★★★★☆

**評価理由:**
ターミナルの 1 行入力欄で複雑なプロンプトを書くのは不便であり、慣れ親しんだエディタ（Vim、Emacs、VS Code 等）でプロンプトを編集できることは生産性を大幅に向上させる。特に長文の指示や構造化されたプロンプトを書く場合に効果大。ヘルプに追加されたことで discoverability も向上。

**具体的なユースケース例:**
- Ctrl+G で Vim を起動し、マルチパラグラフの詳細な実装仕様書をプロンプトとして作成
- `$EDITOR=code` 設定下で VS Code を開き、コードレビュー指示を構造化して記述
- 長いシステムプロンプトのドラフトを繰り返し編集する際の試行錯誤を快速化

---

### 11. AskUserQuestion の「Other」入力フィールドで外部エディタ（Ctrl+G）をサポート

**バージョン:** 2.1.9

**機能の概要:**
Claude がインタラクティブな質問を行う際の「Other」自由入力フィールドで Ctrl+G による外部エディタ起動をサポート。

**採点:** ★★☆☆☆

**評価理由:**
2.1.20 の Ctrl+G ヘルプメニュー追加と連動した機能。インタビュー形式の質問応答シナリオで詳細な回答を書く際に便利。ただし使用頻度は主プロンプト入力より低いため、補完的な改善として ★★☆☆☆。

**具体的なユースケース例:**
- Claude が「どのようなアーキテクチャを想定していますか？」と聞いてきた際、Ctrl+G で Vim を開いて詳細な要件を記述
- 「Other」選択後に複数の条件や制約事項を外部エディタで整理してから送信

### 12. Ctrl+G でエディタ起動失敗時のエラー表示

**バージョン:** 2.1.6

**機能の概要:**
Ctrl+G で外部エディタの起動に失敗した場合、エラーメッセージを表示して原因特定をサポート。

**採点:** ★☆☆☆☆

**評価理由:**
エラーハンドリングの改善であり、ユーザー体験の細かな向上。失敗時に無反応だった状態から原因が分かるようになる点は価値があるが、機能そのものを新しく提供するものではなく既存機能の堅牢化。

**具体的なユースケース例:**
- `$EDITOR` 変数が未設定の場合に「エディタが設定されていません」のエラーを確認
- パスが通っていないエディタを指定していた際の設定ミスを素早く特定

### 13. 音声 STT で 10 の新言語をサポート（合計 20 言語）

**バージョン:** 2.1.69

**機能の概要:**
音声入力（STT）で新たにロシア語、ポーランド語、トルコ語、オランダ語、ウクライナ語、ギリシャ語、チェコ語、デンマーク語、スウェーデン語、ノルウェー語を追加。合計 20 言語をサポート。

**採点:** ★★★☆☆

**評価理由:**
非英語圏の開発者にとって、母国語での音声入力は思考速度でプロンプトを入力できる体験を実現する。ただし対象ユーザーが特定言語話者に限定されるため、全体的なインパクトは限定的。対象言語話者には大きな価値を持つ。

**具体的なユースケース例:**
- ロシア語話者の開発者が母国語で「APIエンドポイントを設計してください」と音声入力
- スウェーデン語のコメントを含むコードレビューを、スウェーデン語で音声指示しながら実施

### 14. `voice:pushToTalk` キーバインディングで音声アクティベーションキーを再バインド可能

**バージョン:** 2.1.71

**機能の概要:**
音声入力のプッシュトゥトーク機能のアクティベーションキーを `keybindings.json` で変更可能。デフォルトは `space`。

**採点:** ★★★☆☆

**評価理由:**
スペースキーはプロンプト入力中に誤って音声入力をトリガーしやすく、特に IME 使用環境では既存の日本語変換確定と競合する問題があった。アクティベーションキーを変更できることで、ユーザーの環境に合わせた快適な音声入力体験が実現する。

**具体的なユースケース例:**
- 日本語 IME ユーザーがスペースキーを避けて `Ctrl+Space` や `F1` に音声入力を再バインド
- `keybindings.json` に `{"action": "voice:pushToTalk", "key": "ctrl+shift+v"}` を追加
- マウスのサイドボタンに相当するキーをトリガーに設定してハンズフリー操作を実現

---

### 15. 日本語 IME からの全角スペース入力をチェックボックス選択でサポート

**バージョン:** 2.1.31

**機能の概要:**
日本語 IME を使用中に全角スペースを入力した場合でも、チェックボックスの選択操作として認識されるようになった。

**採点:** ★★★☆☆

**評価理由:**
日本語ユーザーが Claude Code を使う上で慢性的な問題だった IME との干渉を解消する。全角スペースは日本語入力環境で誤入力しやすく、これが認識されないとチェックボックス操作のたびに IME を切る必要があった。日本語ユーザーには使い勝手が大幅に向上するが、対象ユーザーが限定される。

**具体的なユースケース例:**
- 許可ダイアログで日本語 IME をオンにしたまま全角スペースで「はい」を選択
- 複数選択プロンプトで IME 状態を気にせず快適にチェックボックス操作

---

### 16. オプション選択プロンプトで日本語 IME からの全角数字入力をサポート

**バージョン:** 2.1.21

**機能の概要:**
番号選択プロンプトで日本語 IME が有効な状態で入力した全角数字（１、２、３…）がオプション番号として正しく認識される。

**採点:** ★★★☆☆

**評価理由:**
2.1.31 の全角スペース対応と同様に、日本語ユーザーの IME 干渉問題を解消する。番号入力のたびに IME を切り替える必要がなくなり、会話の流れを妨げない。日本語ユーザーへのインパクトは大きいが、対象ユーザーが限定される。

**具体的なユースケース例:**
- モデル選択ダイアログで IME オンのまま「１」を入力して最初のオプションを選択
- スキル選択メニューで全角数字で直接番号入力してナビゲーション

### 17. プロンプトフッターに PR レビューステータスインジケーター表示

**バージョン:** 2.1.20

**機能の概要:**
プロンプトフッター（ステータスライン）に現在のブランチに関連する PR のレビューステータス（承認数・変更要求等）をリアルタイムで表示。

**採点:** ★★★★☆

**評価理由:**
Claude Code を使いながら PR の状況を確認するためにブラウザや別ツールに切り替える必要がなくなる。コーディングの文脈を保ったまま PR ステータスを把握できることは、レビュー待ちのタスクを効率よく管理する上で非常に有用。特に複数 PR を並行で進める開発者には大きなメリット。

**具体的なユースケース例:**
- フッターに「PR #123: ✓2 ✗1」と表示され、変更要求があることを確認しながら修正作業を継続
- PR のレビュー承認が揃ったタイミングをプロンプトフッターで検知してマージ操作に移行
- Claude Code で作業中に他の PR のレビューステータスが変わったことを視覚的に把握

---

### 18. エフォートレベル表示をロゴとスピナーに追加

**バージョン:** 2.1.69

**機能の概要:**
現在設定されているモデルのエフォートレベル（例: "with low effort"）をスピナーとロゴエリアに表示。エフォートレベルが何に設定されているかを常時把握可能。

**採点:** ★★☆☆☆

**評価理由:**
`/effort` でエフォートレベルを変更した後、現在の設定を都度確認する手間が省ける UI 改善。特に複雑なタスクと簡易タスクを切り替えながら使う場合に誤ったエフォート設定で作業するミスを防ぐ。マイナーだが実用的な改善。

**具体的なユースケース例:**
- ロゴ部分に「Claude (low effort)」と表示され、節約モードで動作中であることを確認
- high effort に設定してコード生成中、スピナーで現在のエフォートレベルを確認して想定通りか検証

---

### 19. `spinnerTipsOverride` 設定でスピナーのヒントをカスタマイズ可能

**バージョン:** 2.1.45

**機能の概要:**
Claude が処理中に表示するスピナーのヒントテキストを `settings.json` の `spinnerTipsOverride` でカスタマイズ可能。

**採点:** ★☆☆☆☆

**評価理由:**

### 22. 空の bash プロンプトで Ctrl+U により bash モードを終了

**バージョン:** 2.1.69

**具体的なユースケース例:**
- bash モードでコマンドを実行した後、何も入力せず `Ctrl+U` で Claude との会話モードに即座に戻る
- bash モードに誤って入ってしまった際に素早く抜け出す
